# アカウント認証

## 認証方式

quicklog ではアカウント認証に Supabase Auth を使用しています。
現時点ではメールアドレス・パスワード認証にのみ対応しています。

将来的には Google アカウントを使ったソーシャルサインインに対応する予定です。

## セッション管理情報

quicklog では、Supabase Auth の `session` にアプリ固有のデータスコープ情報を組み合わせてセッション状態を管理しています。
これは `session` の状態とローカルデータの保存先が必ずしも一対一対応しないためです。

たとえばネットワークエラーが発生してセッションの取得に失敗した場合、直前までサインイン中だったユーザー情報を保持しておいて、そのユーザーのローカルデータに継続して書き込めるようにします。

以下で quicklog で扱うセッション管理情報を説明します。

### Supabase Auth session

Supabase Auth から取得する認証セッションです。`session` が存在する場合、Supabase 上ではユーザー認証が済んでいます。これはクラウド同期を行うための必要条件です。

`session` が存在しない場合、その理由は複数考えられます。

- まだサインインしていない
- 明示的にサインアウトした
- 期限切れなどでセッションが失われた
- ネットワークエラーなどでセッションを取得できなかった

意図しないセッション喪失が起こってもユーザーが継続的に書き込みを行えるよう、quicklog では `session` 以外のデータスコープ情報を組み合わせて表示・保存先データを決定します。

### StoredDataScope

`StoredDataScope` は localStorage に保存するデータスコープ情報です。アプリ起動時に localStorage からこの値を読み込むことで、前回の終了時に匿名状態 (`anonymous`) であったか、特定のユーザーとしてサインインしていたか (`user`) を判別します。
特にデータスコープが `user` であった場合、ローカルデータの保存先を決定するために `userId` の情報を持ちます。

### RuntimeSessionState

`RuntimeSessionState` はユーザーがどの状態にあり、quicklog がどのデータを表示・保存・同期の対象として扱うかを表す状態です。

- `anonymous`: 匿名状態を表し、`quicklog.data.anonymous` を扱う
- `authenticated`: ユーザーとしてサインイン済みの状態を表し、`quicklog.data.user.<user-id>` を扱い、クラウド同期も行う
- `authPending`: 前回のデータスコープがユーザーであり、Supabase Auth から `session` を取得・確認している途中の状態を表す。`quicklog.data.user.<user-id>` を扱うが、クラウド同期はまだ行えない
- `sessionLost`: 直前までサインイン中だったユーザーについて、有効な `session` を取得できない状態を表す。`quicklog.data.user.<user-id>` を扱うが、クラウド同期は行えない

## セッション状態の解決

### アプリ起動時

アプリ起動時に localStorage から `StoredDataScope` を読み込みます。

`StoredDataScope` が `anonymous` であった場合は `RuntimeSessionState` も `anonymous` に設定して匿名データを読み込みます。
`StoredDataScope` が `user` であった場合は Supabase Auth から `session` を取得します。

`session` 状態の確認中は `RuntimeSessionState` を `authPending` に設定し、認証中でもローカルのユーザーデータに書き込みを許します。

`session` が取得できなかった場合、`RuntimeSessionState` を `sessionLost` に設定し、ユーザーに再びサインインするよう促します。

有効な `session` が取得できた場合は `RuntimeSessionState` を `authenticated` に設定し、クラウド同期を行います。

### アカウント作成時

アカウント作成を試みた場合、Supabase Auth の OTP 認証機能を用いてメールアドレス保有者であることを確認し、その後すぐにサインインを行います。

アカウント作成後にサインインするため、quicklog はアカウント作成時に記入されたパスワードを一時的に保持します。メールアドレス確認後のサインイン成功、認証のキャンセル、リロードのいずれかによってパスワードは破棄されます。

匿名状態のデータの扱いについては下記のサインイン時の項を参照してください。

### サインイン時

サインインに成功した場合、quicklog はデータスコープをサインインしたユーザーのものへ切り替え、`StoredDataScope` に `user` としてデータスコープ情報を書き込みます。

匿名状態のデータが残っている場合は、まずサインインしたユーザーのローカルデータへマージし、匿名状態のデータを削除した後にクラウド同期を行います。

匿名状態のデータをユーザーデータにマージしたくない場合、ユーザーはデータ管理パネルから匿名データを削除する必要があります。

### サインアウト時

明示的にサインアウトした場合、サインイン中のユーザーのデータは localStorage に残したまま、データスコープだけを匿名データに差し替えて `StoredDataScope` に `anonymous` としてデータスコープ情報を書き込みます。

サインアウトはアカウントやクラウド上のデータを削除する操作ではありません。

### セッション取得に失敗した場合

ネットワークエラーによってセッションを取得できない場合でも、quicklog はデータスコープを匿名状態へ自動で切り替えません。

直前までサインイン中だったユーザーの情報が残っている場合、サインイン中ユーザーのローカルデータを扱いますが、クラウド同期はできない状態として扱います。

この状態で記録したデータは `quicklog.data.user.<user-id>` に保存され、次にクラウド同期に成功したときに反映されます。

## パスワードの再設定と変更

### パスワードを忘れた場合

サインインしていない状態でパスワードを再設定するには、Supabase Auth の OTP 認証機能を用いてメールアドレスの保有者であることを確認し、一時セッション上で新しいパスワードを設定します。

一時セッションはパスワードの変更の完了またはキャンセル時に破棄され、データスコープは匿名状態に戻ります。アカウントを継続して利用するには再度サインインが必要です。

### サインイン中のパスワード変更

サインイン中にパスワードを変更するには、現在のパスワードと新しいパスワードを記入し、現在のパスワードでサインイン検証を行った後にパスワードを変更します。

## アカウントの削除

ユーザーがアカウントの削除を行うには、まずサインインして有効な `session` を取得する必要があります。
アカウントの削除操作は Supabase の Edge Function を通して行われるため、認証されていないユーザーや、別のユーザーのアカウントを削除することはできません。

ユーザーデータの削除を行う直前に、ユーザーのクラウド上のデータをローカルデータと同期します。
次いでローカルデータを匿名データに退避し、ユーザーのアカウント情報とクラウド上の同期データを削除します。
最後に `quicklog.data.user.<deleted-user-id>` を localStorage から削除し、セッション情報をクリアします。
これは「クラウド同期アカウント」のみの削除を行い、端末上ではメモを継続して扱えるようにするための措置です。

正常なフローを辿っている場合、削除操作の実行時に匿名データは空であるはずです。もし匿名データが空でない場合、アカウント削除操作の開始は拒否されます。

## 既知の制約

- 現時点で quicklog 本番環境は Supabase 標準 SMTP に依存しており、認証メールの送信数に強い制限があります。
- メールアドレス・パスワード認証のみ実装済みで、外部サービスを使ったサインインには対応していません。
- このことから、本番環境を不特定多数に継続して公開する場合、カスタム SMTP の設定が必要になります。
